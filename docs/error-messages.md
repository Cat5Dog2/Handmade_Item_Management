# エラーメッセージ設計書（error-messages）

## 1. 目的

本書は、ハンドメイド在庫・販売管理アプリのMVPにおける**画面表示用エラーメッセージの既定値**を定義する文書である。  
API の `code` / `message` / `details` を、画面側でどのような日本語メッセージへ変換し、どこへ表示し、利用者に何を促すかを統一する。

本書の主目的は次のとおり。

- 画面ごとに文言の揺れを防ぐ
- Codex / 実装者が迷わずフロント実装できるようにする
- API エラーコードと画面導線を一対一で近づける
- 項目エラー、画面上部エラー、トースト、ダイアログの使い分けを明確にする

---

## 2. 文書の位置づけ

仕様衝突時の優先順位は次のとおりとする。

1. `docs/requirements.md`
2. `docs/basic_design.md`
3. `docs/detail_design.md`
4. `docs/api_specification.md`
5. `docs/data_design.md`
6. `docs/screen_design.md`
7. `docs/implementation-notes.md`
8. 本書 `docs/error-messages.md`

本書は、上位設計を前提に**文言・表示位置・画面導線を固定する補足文書**として扱う。

## 3. 基本方針

### 3.1 文言方針

- 文体は丁寧語で統一する
- 1文はできるだけ短くする
- 利用者が次に取る行動を想像しやすい文言にする
- 開発者向けの用語や内部コードは画面にそのまま出さない
- 可能な限り「何が悪いか」より「どう直せばよいか」を先に伝える

### 3.2 表示優先順位

エラーの表示優先順位は次のとおりとする。

1. 項目単位で特定できるものは**入力欄の近傍**に表示する
2. フォーム全体に関わるものは**画面上部アラート**に表示する
3. 再読み込み・再取得が前提のものは**画面内エリアエラー**で表示する
4. 遷移直前の通知や非ブロッキング通知は**トースト**で表示する
5. 削除や状態変更前の確認は**ダイアログ**で表示する

### 3.3 文言の禁止事項

- API が返した英語メッセージの露出
- スタックトレースや例外名の露出
- 「不明なエラー」だけで終わる案内
- 同じ内容のメッセージを項目欄と画面上部に重複表示すること

## 4. 表示位置の使い分け

| 区分 | 用途 | 主な対象 |
|---|---|---|
| 項目エラー | 入力値の誤りを修正してもらう | 必須不足、文字数超過、形式不正 |
| 画面上部アラート | 項目にひも付けにくい失敗を知らせる | 参照先不正、重複、更新不可 |
| 画面内エラー | 取得失敗や対象消失を知らせる | 一覧取得失敗、詳細取得失敗 |
| トースト | 遷移や再試行を妨げない通知 | セッション切れ、保存失敗、再試行案内 |
| ダイアログ本文 | 実行前確認または実行不能理由の補足 | 削除確認、販売済更新確認 |

### 4.1 `details` の扱い

- API 応答に `details` がある場合は、まず項目エラーへマッピングする
- 同一項目に複数エラーがある場合は、原則として最初の1件のみ表示する
- `details` の項目名が画面の入力名と一致しない場合は、フロント側で変換テーブルを持つ
- `details` が無い場合は `code` ベースの共通メッセージを表示する

### 4.2 401 / 403 の扱い

- `AUTH_REQUIRED` はトーストまたは画面メッセージで「セッションが切れました。再度ログインしてください。」を表示し、`/login` へ遷移する
- `AUTH_FORBIDDEN` は「この操作は実行できません。」を表示し、`/login` へ遷移する
- 401 / 403 は項目エラーにしない

## 5. APIエラーコード共通マッピング

### 5.1 共通マスター

| code | HTTP | 既定メッセージ | 推奨表示位置 | 利用者への案内 |
|---|---:|---|---|---|
| `AUTH_REQUIRED` | 401 | セッションが切れました。再度ログインしてください。 | トースト + 画面遷移 | ログイン画面へ戻す |
| `AUTH_FORBIDDEN` | 403 | この操作は実行できません。 | トースト + 画面遷移 | ログイン画面へ戻す |
| `VALIDATION_ERROR` | 400 | 入力内容を確認してください。 | 項目エラー or 画面上部 | 入力値を見直して再実行 |
| `PRODUCT_NOT_FOUND` | 404 | 対象の商品が見つかりません。 | 画面内エラー | 一覧へ戻る / 再検索 |
| `PRODUCT_DELETED` | 404 | 対象の商品はすでに利用できません。 | 画面内エラー | 一覧へ戻る |
| `PRODUCT_RELATED_RESOURCE_UNAVAILABLE` | 404 | この商品の関連情報は表示できません。 | 画面内エラー | 商品一覧へ戻る |
| `CATEGORY_NOT_FOUND` | 400 / 404 | 指定したカテゴリが見つかりません。 | 項目エラー or 画面内エラー | 一覧を再取得して選び直す |
| `TAG_NOT_FOUND` | 400 / 404 | 指定したタグが見つかりません。 | 項目エラー or 画面内エラー | 一覧を再取得して選び直す |
| `TASK_NOT_FOUND` | 404 | 対象のタスクが見つかりません。 | 画面上部 or 画面内エラー | 一覧を再取得する |
| `CUSTOMER_NOT_FOUND` | 404 / 400 | 対象の顧客が見つかりません。 | 画面内エラー or 画面上部アラート | 一覧へ戻る / 再検索 / 再選択 |
| `CUSTOMER_ARCHIVED` | 400 | アーカイブ済みの顧客は現在利用できません。 | 画面上部アラート | 詳細画面で内容を確認する / 別の顧客を選び直す |
| `CATEGORY_IN_USE` | 400 | 使用中のカテゴリは削除できません。 | ダイアログ内 or 画面上部 | 参照中の商品を確認する |
| `TAG_IN_USE` | 400 | 使用中のタグは削除できません。 | ダイアログ内 or 画面上部 | 参照中の商品を確認する |
| `DUPLICATE_NAME` | 400 | 同じ名前は登録できません。 | 項目エラー | 別の名前を入力する |
| `IMAGE_LIMIT_EXCEEDED` | 400 | 画像は最大10枚まで登録できます。 | 画面上部 | 不要な画像を削除してから追加 |
| `UNSUPPORTED_IMAGE_TYPE` | 400 | JPEG、PNG、WebP 形式の画像を選択してください。 | 画面上部 | 対応形式へ変更して再実行 |
| `IMAGE_TOO_LARGE` | 400 | 画像サイズが大きすぎます。10MB以下の画像を選択してください。 | 画面上部 | 画像を圧縮して再実行 |
| `IMAGE_NOT_FOUND` | 404 | 対象の画像が見つかりません。最新の情報を読み込み直してください。 | 画面上部 | 商品詳細を再取得する |
| `INVALID_STATUS_FOR_QR_SELL` | 400 | このステータスの商品はQRで販売済に更新できません。 | 結果画面 or ダイアログ | 商品状態を確認する |
| `ALREADY_SOLD` | 400 | この商品はすでに販売済みです。 | 結果画面 or ダイアログ | そのまま結果表示 |
| `INTERNAL_ERROR` | 500 | 予期しないエラーが発生しました。時間をおいて再度お試しください。 | 画面上部 or 画面内エラー | 再試行導線を表示 |

### 5.2 文脈別の補正ルール

同じ `code` でも文脈に応じてメッセージを微調整してよい。

- `CATEGORY_NOT_FOUND`
  - 商品登録/編集時: 「指定したカテゴリが見つかりません。カテゴリを選び直してください。」
  - カテゴリ編集/削除時: 「対象のカテゴリが見つかりません。最新の一覧を確認してください。」
- `TAG_NOT_FOUND`
  - 商品登録/編集時: 「指定したタグが見つかりません。タグを選び直してください。」
  - タグ編集/削除時: 「対象のタグが見つかりません。最新の一覧を確認してください。」
- `VALIDATION_ERROR`
  - `details` がある場合は共通メッセージを補助的にのみ使う
  - `details` がない場合のみ画面上部に「入力内容を確認してください。」を表示する
- `CUSTOMER_NOT_FOUND`
  - 商品登録/編集時: 「選択した顧客が見つかりません。顧客を選び直してください。」
  - QR販売済更新時: 「選択した顧客が見つかりません。顧客一覧を確認してください。」
  - 顧客詳細時: 「対象の顧客が見つかりません。」を表示し、一覧へ戻る導線を表示する
- `CUSTOMER_ARCHIVED`
  - 商品登録/編集時: 「選択した顧客は現在利用できません。別の顧客を選び直してください。」
  - QR販売済更新時: 「選択した顧客は現在利用できません。別の顧客を選択してください。」
  - 顧客編集時: 「アーカイブ済みの顧客は編集できません。詳細画面で内容をご確認ください。」
  - 顧客詳細時: 参照は許可し、編集導線は非表示または非活性とする

## 6.2 ダッシュボード（SCR-02）

| 条件 | メッセージ | 表示位置 |

| 取得失敗 | ダッシュボード情報を取得できませんでした。 | 画面内エラー |

- ダッシュボード全体が0件でも、件数カードは `0` 表示を維持する
- 空状態メッセージは、リスト表示領域ごとに個別に出し分ける

## 7. 通信失敗・想定外エラーの扱い

## 7.1 通信失敗

通信断、タイムアウト、5xx 相当、CORS 失敗などで API の `code` を受け取れない場合は、次の文言を既定とする。

- 取得系: 「データを取得できませんでした。通信状況を確認して再度お試しください。」
- 登録/更新系: 「保存に失敗しました。通信状況を確認して再度お試しください。」
- 削除系: 「削除に失敗しました。通信状況を確認して再度お試しください。」
- QR系: 「通信に失敗しました。再度お試しください。」

## 7.2 `INTERNAL_ERROR`

- 基本文言: 「予期しないエラーが発生しました。時間をおいて再度お試しください。」
- 再試行可能な取得系画面では「再試行」ボタンを表示する
- 送信系では、同一操作をむやみに多重実行させないため、送信中解除後に明示的な再実行で再送する

## 8. フロント実装ルール

### 8.1 実装単位

フロントでは少なくとも以下を分離する。

- `errorCodeMessages.ts`: `code` → 既定文言
- `fieldErrorMessages.ts`: 項目別・画面別文言
- `mapApiErrorToUi.ts`: API エラーを UI エラーへ変換
- `normalizeApiDetails.ts`: `details` をフォーム項目へ寄せる

### 8.2 実装ルール

- API の `message` を無条件でそのまま表示しない
- 表示文言は基本的にフロント側の定義を正本とする
- API の `message` はログ出力や開発確認には使ってよいが、利用者向け表示は本書の文言を優先する
- 画面固有の文脈がある場合は、共通メッセージをそのまま使わず画面固有文言へ上書きしてよい
- `PRODUCT_NOT_FOUND` などの 404 は、単なるトーストで終わらせず、戻り導線または再取得導線を表示する

### 8.3 `details.field` の推奨キー

MVPでは次のキー名を推奨する。

- `email`
- `password`
- `keyword`
- `name`
- `description`
- `price`
- `categoryId`
- `tagIds`
- `status`
- `primaryImageId`
- `file`
- `content`
- `dueDate`
- `memo`
- `qrCodeValue`

フロントはこのキーを画面ごとの入力名へ変換して扱う。
