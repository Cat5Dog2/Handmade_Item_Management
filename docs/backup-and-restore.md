# バックアップ / 復旧手順書（backup-and-restore）

## 1. 目的

本書は、ハンドメイド在庫・販売管理アプリのMVPにおける**業務データと画像データの保全方針、および障害時の復旧手順**を定義するための運用文書である。  
設計書で「別途運用設計で定義する」としたバックアップ / 復旧手順を、実運用できる粒度まで具体化することを目的とする。

本書の主対象は次の2つである。

- Firestore に保存される業務データ
- Cloud Storage に保存される商品画像

あわせて、復旧時に参照すべき補助情報として `operationLogs` を扱う。

---

## 2. 適用範囲

本書はMVPの以下の構成に適用する。

- フロントエンド: Firebase Hosting
- API: Cloud Run
- 認証: Firebase Authentication
- データベース: Firestore
- 画像保存: Cloud Storage

本書は**アプリケーションデータ保全**を主対象とし、次は主対象外とする。

- `node_modules` など再生成可能なビルド生成物
- ローカル開発環境の一時ファイル
- 期限付きURLそのもの
- Firebase Authentication のアカウント棚卸し手順の詳細
- 個々の開発端末のローカル設定

補足:

- 期限付きURLは復旧対象ではない。`displayPath` / `thumbnailPath` からAPIが再生成する。
- `firebase.json`、rules、indexes、設計書、ソースコードは**Git リポジトリを正本**とする。
- シークレット値や本番環境変数は、別途安全な運用手段で管理する。本書では値そのものの保管手順までは定義しない。

---

## 3. 基本方針

## 3.1 保全の考え方

MVPでは、次の3層で保全する。

1. **Firestore の定期エクスポート**
2. **Cloud Storage の削除・上書き保護**
3. **運用ログによる影響範囲特定**

このため、復旧は単に「全部戻す」ではなく、以下の順で行う。

1. 影響範囲を把握する
2. 書き込みを止める
3. 安全な場所で復旧データを確認する
4. 本番へ必要最小限を戻す
5. 再発防止と証跡整理を行う

## 3.2 復旧の原則

- **いきなり本番へ全面インポートしない**
- まずは別の検証環境または一時環境で復旧データを確認する
- Firestore のインポートは既存ドキュメントを上書きしうるため、無差別な再投入を避ける
- `operationLogs` と現在値を照合して、必要な復旧対象だけを切り出す
- 障害発生中は API の更新系を止め、状態を固定してから作業する

## 3.3 MVPにおける割り切り

- 単一プロジェクト運用を前提とする
- 本番DBのポイントインタイム復旧までは前提にしない
- 画像の広域冗長化や別プロジェクト退避は将来拡張とする
- まずは**誤削除・誤更新・限定的なデータ破損から戻せること**をMVPの到達点とする

---

## 4. 保護対象データ

## 4.1 Firestore

主な対象コレクションは次のとおりとする。

- `products`
- `tasks`
- `categories`
- `tags`
- `counters`
- `operationLogs`

補足:

- `products` は論理削除を含む業務中核データである
- `tasks`、`categories`、`tags` は物理削除がありうるため、誤操作復旧対象に含める
- `counters/product` は採番整合性に影響するため、復旧時に必ず確認する
- `operationLogs` は調査・整合確認のため重要だが、ログ自体の欠損より業務データ復旧を優先する

## 4.2 Cloud Storage

主な対象は商品画像とする。

- `products/{productId}/display/...`
- `products/{productId}/thumb/...`

補足:

- 元画像はMVPでは保持しない前提である
- 復旧対象は表示用画像とサムネイル画像とする
- 復旧後の画面表示は signed URL 再生成で回復させる

## 4.3 Git 管理ファイル

次はバックアップ対象というより、**Git を正本として再配置する対象**である。

- `firebase.json`
- `.firebaserc`
- `firebase/firestore.rules`
- `firebase/firestore.indexes.json`
- `firebase/storage.rules`
- `AGENTS.md`
- `README.md`
- `docs/*.md`
- アプリケーションソースコード

---

## 5. 目標値

MVPでは、厳密なSLAではなく運用目安として次を置く。

| 項目 | 目標 |
|---|---|
| Firestore 定期バックアップ頻度 | 1日1回以上 |
| Firestore 想定RPO | 24時間以内 |
| Firestore 想定RTO | 同営業日内を目安 |
| 画像誤削除の一次復旧 | 数時間以内を目安 |
| 重大障害時の業務再開 | 1営業日以内を目安 |

補足:

- RPO は「どこまでデータ損失を許容するか」の目安である
- RTO は「どれくらいで復旧させるか」の目安である
- 単独運用のMVPとして無理のない水準を採用する

## 6. バックアップ方針

## 6.1 Firestore バックアップ方針

Firestore は**定期的に Cloud Storage へエクスポート**する。  
エクスポート対象は原則として**データベース全体**とする。

理由:

- コレクション間の参照関係を分断しにくい
- 事故時に「どこまで戻せるか」を説明しやすい
- `products` だけでなく `tasks`、`categories`、`tags`、`counters` の整合を取りやすい

### 6.1.1 保存先

Firestore バックアップ用に専用の Cloud Storage バケットまたは専用プレフィックスを用意する。

推奨例:

```text
gs://<backup-bucket>/firestore/YYYY/MM/DD/HHmmss/
```

### 6.1.2 実行頻度

MVPの既定は次のとおりとする。

- 定期バックアップ: 毎日1回
- 実行時刻: 深夜帯または利用が少ない時間帯
- 追加バックアップ: 大きなリリース前、データ移行前、手動大量更新前

### 6.1.3 保持期間

MVPでは次を推奨する。

- 日次バックアップ: 35日保持
- 月次代表バックアップ: 12か月保持

月次代表バックアップは、月初または月末の1世代を残す運用でよい。

### 6.1.4 注意事項

- エクスポートは実行中の変更を完全な一点時刻スナップショットとして保証しない前提で扱う
- バックアップ時の読み取り課金と保存コストを見込む
- 復旧用バケットには不要な一般公開設定をしない

## 6.2 Cloud Storage バックアップ方針

商品画像は、MVPでは**オブジェクト保護 + 運用復元**で保全する。

基本方針:

- バケットの soft delete を有効にしたまま運用する
- Object Versioning を有効化する
- 古い版を無期限保持せず、ライフサイクルで整理する

### 6.2.1 推奨設定

- soft delete: 有効
- retention duration: 既定値または組織方針に合わせる
- Object Versioning: 有効
- noncurrent object の自動削除: 30日程度で整理

### 6.2.2 この方式で守れるもの

- 画像の誤削除
- 画像の誤上書き
- 一時的な運用ミスによる最新版喪失

### 6.2.3 この方式で弱いもの

- プロジェクト全体削除
- 同一権限者による広域破壊
- 長期間経過後に気づいた画像欠損

そのため、MVPではまず短中期の運用事故を防ぎ、将来必要になれば別プロジェクト退避や定期複製を追加する。

## 6.3 Git 管理物の保全方針

運用設定・設計書・実装コードは Git リポジトリを正本とする。  
障害時は Git から再取得し、再デプロイする。

対象例:

- フロント / API ソースコード
- Firebase rules / indexes
- 設計書、補足文書
- テストコード

## 6.4 手動バックアップを追加で行う場面

次の作業前には、定期バックアップとは別に**手動バックアップ**を取得する。

- 大量のステータス更新
- データ移行スクリプト実行
- 一括画像差し替え
- コレクション構造変更
- 本番での障害調査のために一時的な修正を入れる前

## 7. 役割分担

| 役割 | 主な責務 |
|---|---|
| 運用担当 | バックアップ実行確認、障害一次切り分け、復旧実施判断 |
| 開発担当 | 復旧スクリプト準備、整合確認、再発防止対応 |
| 利用者本人 | 障害発見時の申告、業務影響の確認 |

MVPでは同一人物が兼務してよいが、作業時は**判断・実行・確認を意識的に分ける**。

---

## 8. 平常時の運用手順

## 8.1 初期設定

初回セットアップ時に少なくとも次を確認する。

1. Firestore エクスポート先バケットがある
2. Firestore エクスポート権限が正しく付与されている
3. 画像バケットで soft delete が有効である
4. 画像バケットで Object Versioning が有効である
5. ライフサイクルルールが設定されている
6. バックアップ実行担当者を決める
7. 実行結果を記録する場所を決める

## 8.2 定期確認項目

最低でも週1回、次を確認する。

- Firestore バックアップが予定どおり作成されている
- 最新バックアップの日時が把握できる
- バックアップ容量の異常増減がない
- 画像バケットの保護設定が意図せず変更されていない
- `operationLogs` が継続出力されている

## 8.3 リリース前確認

本番反映前に次を確認する。

- 最新の日次バックアップが存在する
- 大きな変更なら直前の手動バックアップも取得した
- 復旧に必要な Git リビジョンを特定できる
- 変更対象API / バッチ / スクリプトの影響範囲を説明できる

---

## 9. 障害時の共通手順

## 9.1 初動

障害・誤操作を検知したら、次の順で対応する。

1. 発生日時を記録する
2. 症状を記録する
3. 影響範囲を記録する
4. 更新系処理を止める
5. `operationLogs`、対象画面、APIログを確認する
6. 直近バックアップの有無を確認する
7. 復旧方式を決める

## 9.2 更新系停止の方法

状況に応じて次のいずれかを行う。

- Cloud Run の更新系エンドポイントへの利用を止める
- 一時的にメンテナンス連絡を出し、操作を停止する
- 必要ならデプロイを直前版へ戻す

MVPでは利用者が本人中心であるため、**まず操作を止めて状態固定すること**を優先する。

## 9.3 影響範囲の確認観点

- どのコレクションに影響したか
- 単一ドキュメントか複数ドキュメントか
- 画像ファイルも影響したか
- 誤削除か誤更新か重複投入か
- 障害は継続中か、すでに止まっているか
- 直近で実行した手動操作・一括処理・デプロイは何か

---

## 10. Firestore 復旧手順

## 10.1 基本原則

Firestore の復旧は、原則として次のどちらかで行う。

1. **一時環境へインポートして対象データを比較し、必要分だけ戻す**
2. **限定的な影響ならエクスポート内容を参照して手動またはスクリプトで再反映する**

本番への全面インポートは最後の手段とする。

## 10.2 単一商品の誤更新 / 誤削除

対象例:

- 商品名や価格を誤更新した
- `isDeleted=true` にしてしまった
- 販売済更新を誤って実行した

手順:

1. 影響した `productId` を特定する
2. `operationLogs` から変更者・変更時刻・変更概要を確認する
3. 直近バックアップを選ぶ
4. 一時環境へ復旧データを展開し、対象商品の正しい内容を確認する
5. 本番の現行値と差分を比較する
6. 必要な項目だけを API または管理用スクリプトで修正する
7. 修正後に一覧 / 詳細 / ダッシュボード整合を確認する
8. `operationLogs` に復旧実施内容を記録する

注意:

- `updatedAt`、`soldAt`、`status` の整合を必ず確認する
- 誤って `counters` を巻き戻さない
- signed URL は再取得で再生成されるため、そのまま戻さなくてよい

## 10.3 複数商品の誤更新 / 一括破損

対象例:

- 一括処理で複数商品のステータスを壊した
- マイグレーションスクリプトで広範囲に誤更新した

手順:

1. 影響期間を確定する
2. 影響したコレクションと件数を見積もる
3. 最新正常バックアップを選ぶ
4. 一時環境へインポートして正データを抽出する
5. 差分一覧を作る
6. 差分適用スクリプトを用意する
7. 本番更新を停止した状態で差分反映する
8. 代表ケースを画面とAPIの両方で確認する
9. 必要なら再度差分補正を行う
10. 復旧記録を残す

全面インポートを検討してよい条件:

- 影響範囲が全体に近い
- 手動差分反映より全面戻しの方が安全
- 業務停止を受け入れられる
- 事前に現行データの退避がある

## 10.4 `tasks` / `categories` / `tags` の復旧

- `tasks` は物理削除のため、誤削除時はバックアップからの再投入を基本とする
- `categories` / `tags` は参照整合に注意し、関連 `products` の表示影響も確認する
- 名称変更の誤りは、現行本番での差分修正が優先でよい

## 10.5 `counters` の復旧

`counters/product` は慎重に扱う。

原則:

- **過去値へ安易に戻さない**
- 現在発番済みの最大 `productId` と矛盾しない値へ補正する

確認観点:

- すでに採番済みの商品IDの最大値
- 復旧対象期間中に重複採番が起きていないか
- `nextNumber` または同等項目が将来採番と衝突しないか

## 11. 画像復旧手順

## 11.1 単一画像の誤削除 / 誤上書き

手順:

1. 影響した `productId` と `imageId` を特定する
2. Firestore の `images` 配列内メタデータを確認する
3. Cloud Storage 上の対象パスを確認する
4. soft delete または旧版から対象画像を復元する
5. 必要なら display / thumb の両方を戻す
6. 商品詳細APIを再取得し、画面表示を確認する

確認項目:

- `displayPath` と `thumbnailPath` が存在する
- 画像順序が崩れていない
- 主画像フラグが崩れていない
- signed URL 再生成後に表示できる

## 11.2 複数画像の欠損

- まず Storage 側の影響範囲を一覧化する
- 同時に Firestore の画像メタが壊れていないか確認する
- 画像だけ消えたのか、メタも壊れたのかで復旧方法を分ける
- メタが正常なら Storage 側復元を優先する
- メタも壊れていれば、Firestore 復旧と合わせて戻す

## 11.3 画像復旧時の注意

- 画像を戻しても signed URL 自体は以前の値と一致しない
- API再取得で新URLを受け取る前提で確認する
- 元画像を保持しない設計のため、display / thumb のどちらも欠損した場合は復旧可能世代に依存する

---

## 12. 全体障害時の再構築手順

本番環境が大きく壊れた場合は、次の順で再構築する。

1. Git からアプリ設定・コードを取得する
2. Firebase / Cloud Run の基本設定を再確認する
3. Firestore rules / indexes / Storage rules を再反映する
4. Firestore バックアップからデータを戻す
5. 画像バケット側を復元する
6. API をデプロイする
7. フロントをデプロイする
8. ログイン、一覧、詳細、登録、画像表示、QR販売更新を確認する

参照文書:

- `docs/deployment.md`
- `docs/implementation-notes.md`
- `docs/error-messages.md`

## 13. 実行コマンドの扱い

Firestore のエクスポート / インポートは、Google Cloud の管理機能で実施する。  
代表的にはコンソール操作または `gcloud firestore export` / `gcloud firestore import` を用いる。

例:

```bash
gcloud firestore export gs://<backup-bucket>/firestore/2026/03/29/033000 --database='(default)'
```

```bash
gcloud firestore import gs://<backup-bucket>/firestore/2026/03/29/033000 --database='(default)'
```

注意:

- 実行時の正式なコマンド・権限要件は、その時点の公式ドキュメントで確認する
- 特に Storage の復元系コマンドは機能進化の影響を受けやすいため、実施前確認を必須とする

## 14. 復旧後の確認チェックリスト

復旧作業後は、少なくとも次を確認する。

### 14.1 共通

- ログインできる
- ダッシュボードが表示できる
- 商品一覧が表示できる
- 商品詳細が表示できる
- 画像が表示できる
- 更新系APIが期待どおり動く
- 新規登録で採番重複が起きない
- `operationLogs` に障害と復旧の記録が残る

### 14.2 商品系

- 誤更新した商品が正しい値に戻っている
- 誤削除した商品が期待どおり復元できている
- ステータス、販売日、価格、カテゴリ、タグが整合している
- 商品一覧の件数や絞り込み結果に不自然さがない

### 14.3 画像系

- 代表商品でサムネイルと詳細画像が表示できる
- 画像順序が崩れていない
- 画像追加 / 削除 / 差し替えが継続して行える

### 14.4 QR販売系

- QR参照ができる
- 販売済更新可否判定が正しい
- 販売済更新後の一覧 / 詳細 / ダッシュボード反映が正しい

## 15. 定期リストア訓練

バックアップは「取れている」だけでは不十分である。  
最低でも**四半期に1回**、次の訓練を行う。

1. 最新または任意世代の Firestore バックアップを選ぶ
2. 一時環境へ復元する
3. 主要コレクション件数を確認する
4. 代表商品の詳細と画像メタを確認する
5. 必要なら画像復元手順も試す
6. 問題点を `docs/backup-and-restore.md` に反映する

訓練で確認したい観点:

- 権限不足がないか
- 実行手順に曖昧さがないか
- 復旧後の確認項目が足りているか
- 実運用の所要時間が妥当か

## 18. まとめ

MVPのバックアップ / 復旧方針は次に要約できる。

- Firestore は**定期エクスポート**で守る
- 画像は**soft delete + versioning** を基本に守る
- 復旧は**一時環境で確認してから必要最小限を戻す**
- `operationLogs` を使って影響範囲を把握する
- Git 管理物は**Git を正本**として再配置する

この方針により、MVPとして現実的な運用負荷で、誤削除・誤更新・限定障害からの復旧可能性を確保する。
