# ハンドメイド在庫・販売管理アプリ テスト設計書

## 1. 文書概要

### 1.1 目的
本書は、ハンドメイド在庫・販売管理アプリのMVPに対するテスト方針、対象範囲、観点、実施条件、完了判定を定義するものである。  
要件定義書、基本設計書、画面設計書、詳細設計書、API仕様書をもとに、実装内容を過不足なく検証するための共通基準として利用する。

### 1.2 対象リリース
- MVP 初期リリース

### 1.3 対象システム
- フロントエンド: React + TypeScript + Vite
- バックエンド: Node.js + TypeScript + Express
- 認証: Firebase Authentication
- データベース: Firestore
- 画像保存: Cloud Storage
- 実行基盤: Cloud Run
- ホスティング: Firebase Hosting

### 1.4 参照文書
- 要件定義書
- 基本設計書
- 画面設計書
- 詳細設計書
- API仕様書

---

## 2. テストの目的

本テストの目的は以下とする。

- MVP対象機能が要件どおりに動作することを確認する
- 画面・API・データ更新の整合性を確認する
- 正常系だけでなく、入力エラー、業務ルール違反、論理削除済みデータ、認証切れなどの異常系を確認する
- モバイル中心利用を前提とした主要導線、操作性、表示状態、再試行導線を確認する
- 受入条件を満たすことを確認し、リリース可否判断に必要な材料を揃える

## 4. テスト方針

### 4.1 テストレベル

| レベル | 目的 | 主対象 |
|---|---|---|
| 単体テスト | 関数、コンポーネント、バリデーションの局所確認 | フロント部品、APIロジック |
| 結合テスト | 画面-API-DB/Storage の接続確認 | 商品、画像、タスク、QR、認証 |
| 総合テスト | 業務シナリオ全体の確認 | 画面遷移、主要導線、異常系 |
| 受入れ確認 | 要件定義書の受入条件充足確認 | MVP全体 |

### 4.2 テスト種別

| 種別 | 内容 |
|---|---|
| 正常系テスト | 想定どおりの入力 / 操作 / 業務フロー |
| 異常系テスト | 入力不正、認証エラー、業務ルール違反、対象なし |
| 境界値テスト | 最大文字数、価格境界、画像枚数上限、サイズ上限 |
| 状態遷移テスト | ステータス変更、完了/未完了切替、論理削除前後 |
| 表示テスト | ローディング、0件、プレースホルダー、メッセージ、ダイアログ |
| APIテスト | HTTPステータス、レスポンス形式、エラーコード、業務ルール |
| データ整合性テスト | 更新日時、販売日時、完了日時、論理削除、代表画像整合 |
| 非機能確認 | 主要導線のタップ数、主要表示性能、モバイル/PC表示、ログ出力確認 |

### 4.3 重点確認項目
- 商品IDの一意採番と再利用禁止
- 検索キーワード正規化とAND条件絞り込み
- 検索キーワードの入力制約（最大100文字、改行不可、タブ不可、制御文字不可または無害化）
- 販売済変更時 / 販売済から戻し時の `soldAt` 制御
- 論理削除済み商品の参照不可と集計除外
- 画像上限、形式、サイズ、代表画像自動切替、`sortOrder` 詰め直し
- 期限付き画像URLの有効期限切れ後の再取得
- タスク完了/未完了切替時の `completedAt` 制御
- 未使用カテゴリ / タグのみ削除可能であること
- QR販売済更新の対象ステータス制御、重複更新防止、無効QR判定
- 顧客登録・編集・アーカイブと顧客別購入履歴導出
- 商品更新 / QR販売済更新時の顧客紐付け整合
- 主要操作および主要エラーのログ出力
- 401 / 403 / 通信失敗時の再ログイン・再試行導線

## 6. テストデータ方針

### 6.1 テストデータ区分

| 区分 | 内容 |
|---|---|
| 正常データ | 必須項目充足、標準的な商品/カテゴリ/タグ/タスク/顧客 |
| 境界データ | 最大文字数、0円、画像10枚、10MB近傍、検索キーワード100文字 |
| 異常データ | 必須欠落、空白のみ、改行含む単一行項目、小数価格、非対応画像、タブ/制御文字を含む入力、アーカイブ済み顧客指定 |
| 状態データ | 制作前 / 制作中 / 制作済 / 展示中 / 在庫中 / 販売済、顧客あり / 顧客なし / アーカイブ済み顧客 |
| 削除系データ | 論理削除済み商品、未使用カテゴリ、未使用タグ、アーカイブ済み顧客 |
| 運用確認データ | 期限切れ画像URL、ログ確認対象の正常操作 / 異常操作 |

### 6.2 代表データ例
- 商品ID: `HM-000001`, `HM-000002`
- カテゴリ: 使用中 / 未使用の両方
- タグ: 使用中 / 未使用の両方
- タスク: 未完了 / 完了済み / 納期あり / 納期なし
- 顧客: 通常顧客 / 顧客紐付け済み販売済商品を持つ顧客 / 購入履歴なし顧客 / アーカイブ済み顧客
- 画像: 0枚 / 1枚 / 10枚 / 代表画像あり / 代表画像なし / URL有効期限切れ後再取得確認用
- 検索キーワード: 正常文字列 / 100文字 / 101文字 / 改行含む / タブ含む / 制御文字含む
- QR: 有効商品 / 販売済商品 / 制作系商品 / 論理削除済み商品 / 未登録商品
- ログ確認: ログイン、商品更新、商品削除、顧客更新、顧客アーカイブ、QR販売済更新、エラー発生

## 7. テスト実施条件

### 7.1 開始条件
- 要件定義、基本設計、画面設計、詳細設計、API仕様がレビュー済みであること
- 対象環境へ最新ビルドが反映済みであること
- 必要なテストデータが投入済みであること
- 主要APIが疎通可能であること

### 7.2 完了条件
- 重大な欠陥が未解決で残っていないこと
- 受入条件に対する確認結果が記録されていること
- 正常系・異常系・境界値・主要導線の確認が完了していること
- 必要なエビデンスが保存されていること

### 7.3 中断 / 再開条件
- 認証基盤障害、API停止、データ破損などで継続不能な場合は中断する
- 障害復旧後、影響範囲の再確認を行ったうえで再開する

## 8. 欠陥管理方針

### 8.1 重大度

| 重大度 | 定義 | 例 |
|---|---|---|
| Critical | リリース不可。主要機能が利用できない | ログイン不可、商品保存不可、QR販売済更新誤動作 |
| High | 主要業務に大きな支障 | 論理削除商品が参照できる、販売日時不整合 |
| Medium | 代替手段はあるが不具合 | 並び順誤り、メッセージ不整合 |
| Low | 軽微な表示崩れ、文言揺れ | レイアウトずれ、補助文言の誤字 |

### 8.2 記録項目
- 発生日
- 事象概要
- 発生手順
- 期待結果
- 実際結果
- 重大度
- 発生環境
- スクリーンショット / ログ / リクエスト情報
- 修正結果

| NFR-26 | 調査用エラーログ | エラー時に原因調査可能なログが取得できる |

## 13. 受入条件対応方針

本書のテストケースは、要件定義書の受入条件を最低充足ラインとして作成する。  
受入確認では以下を重点確認する。

- 認証: ログイン / ログアウト / パスワード再設定 / 認証失敗表示 / ログアウト後のURLクエリ非復元
- 商品管理: 一意採番、必須入力、検索、絞り込み、並び替え、論理削除除外
- 画像管理: 最大10枚、形式、サイズ、代表画像補完、プレースホルダー、期限付きURL再取得
- ステータス管理: 販売日時設定、販売済戻し確認、販売日時解除
- タスク管理: 登録、編集、削除、完了状態、完了日時、並び順
- カテゴリ / タグ: 一意性、未使用削除
- QR: 更新可否判定、既販売済、無効QR、販売済更新
- ダッシュボード: 集計件数、納期近いタスク、最近更新商品
- ログ: ログイン、商品更新、商品削除、顧客更新、顧客アーカイブ、QR販売済更新、主要エラー
- 共通: バリデーション、再試行、エラー表示、検索キーワード入力制約
